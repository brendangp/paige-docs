Every Paige project comes with a dedicated Postgres database provisioned automatically at project creation. There is nothing to configure ‚Äî your bot code connects to it through the built-in `dbServices.js` service, and you can view and manage the data directly from the Paige dashboard. The database is isolated per project, so tables and data in one project are never accessible to another.

## What the database is used for

Your bot can store and retrieve any data it needs during conversations. Common use cases include:

## System tables

Paige automatically creates and maintains three system tables in every project database:

| Table | Description |
|---|---|
| `contacts` | One row per person who has messaged your bot, identified by phone number, WhatsApp ID, or both |
| `conversations` | One row per conversation, linked to a contact; stores the current state |
| `messages` | Every inbound and outbound message, linked to a conversation |

Useful columns on `contacts`:

| Column | What it's for |
|---|---|
| `number` | Their phone number. Empty for a username-only contact |
| `username` | Their WhatsApp username, if they have one |
| `wa_user_id` | The ID WhatsApp issues for this person, specific to your business |
| `opted_in` | Whether they can be included in a broadcast. Set to `false` automatically when someone replies STOP |
| `on_whatsapp` | Whether the number is actually on WhatsApp. Empty means *not yet known* ‚Äî which is different from `false` |
| `tags` | Your own labels for this contact, for building broadcast segments |

<Note>
  `on_whatsapp` is worth reading carefully. Empty and `false` are not the same: empty means nobody has checked yet ‚Äî normal for freshly imported contacts ‚Äî while `false` means WhatsApp has confirmed the number isn't reachable.
</Note>

<Warning>
  You can't drop the system tables, and in Production you can't edit their rows either. Paige uses them to know who it's talking to and where each conversation got to ‚Äî changing them by hand is how bots start behaving inexplicably.
</Warning>

Your bot's **execution logs** and **broadcast delivery records** also live in the database, but they're deliberately kept out of the Database tab ‚Äî they have their own homes in [Logs](/guides/logs) and the Broadcasts tab, which present them far better than a raw table would.

## Creating and managing tables

You can create custom tables by asking the AI agent, or by hand in **Tools ‚Üí Database**.

### From the dashboard

The dashboard lets you:

- View all tables and browse their data
- Create a new table with custom columns
- Add columns to an existing table
- Insert, edit, and delete rows
- Clear all rows from a table
- Drop a custom table
- Download a table as CSV, Excel, or JSON
- Run SQL directly against your data

See [the Database tab](/guides/database-tab) for how each of those works, and for the difference between your preview data and your live data.

### Via the Code Agent

The [Code Agent](/agents/code-agent) has full access to your database schema and can create and modify tables as part of building your bot. When you describe a feature that requires data storage, the agent will create the necessary tables before writing the bot code that references them.

You can ask it to create tables, add columns, query rows, or seed test data ‚Äî all in plain language, without leaving the chat.

## Supported column types

When creating tables, you can use the following column types:

`text`, `varchar`, `integer`, `bigint`, `numeric`, `boolean`, `date`, `timestamptz`, `uuid`, `jsonb`, `serial`

<Tip>
  Use `jsonb` for flexible, semi-structured data ‚Äî for example, storing a contact's form submission as a single column rather than normalising every field into its own column.
</Tip>

## Reading and writing from bot code

Your bot code connects to the project database automatically ‚Äî no connection string or configuration required. The starter template includes a database service with pre-built helper functions for common operations (looking up contacts, recording conversations, inserting rows), and you can also write direct queries when you need more control.

Ask the AI agent to write the database access code for you. For example:

> "When a user submits the booking flow, save the booking to a `bookings` table with their name, service, date, and phone number."

The agent will create the table if it doesn't exist and write the code to insert the row.

<Note>
  Your bot code always connects to the same database visible in the dashboard. Any rows inserted by the bot appear immediately in the data browser, and any changes you make in the dashboard are visible to the bot on the next query.
</Note>
